# Moneyraham Homepage Design

Build a finished, verifiable homepage rather than a static concept. Use Korean for visible UI unless the user requests another language.

## Workflow

1. Inspect the current project and preserve existing files. Treat named copy, assets, and brand rules as the source of truth.
2. Confirm the deliverable boundary: design only, local preview, deployment, or GitHub publication. Do not publish or overwrite existing material without explicit approval.
3. Read [homepage-workflow.md](references/homepage-workflow.md) before shaping the page. If `ui-ux-pro-max` is available, use its search/design-system guidance to select the structure, palette, typography, and UX checks.
4. For a multi-step build, state the proposed structure and obtain approval before writing implementation files.
5. Implement responsive, accessible interactions. Provide meaningful hover, focus, loading, and mobile states; do not supply a static mockup when an interactive page is requested.
6. Verify the real local result: build/run it, inspect the browser or DOM where available, and test the main CTA/navigation at desktop and mobile widths.
7. When creating a new homepage program, create `History/YYYY-MM-DD-descriptive-name.md` in the project. Record the approved prompt, decisions, files changed, run command, and verification result. Do not put secrets in History.

## Default homepage priorities

- Start from the visitor's goal, the business outcome, and one primary CTA.
- Make hierarchy clear above the fold: value proposition, proof, action.
- Use a deliberate design system: semantic color tokens, Korean-capable typography, spacing rhythm, responsive breakpoints, and visible focus styles.
- State claims carefully. Do not invent reviews, certifications, prices, performance, clients, or legal guarantees.
- Prefer real assets and supplied copy. Mark placeholders clearly when assets or information are missing.
- Keep comparable metrics and chart series visually consistent.

## Personal experience capture

After a completed homepage task, propose additions to [homepage-workflow.md](references/homepage-workflow.md) only when they are concrete and verified, such as a successful design decision, a discovered user preference, or a reproducible validation step. Do not add vague opinions or customer data.
