# Homepage workflow reference

## Intake

Collect or state assumptions for:

- Brand name, offering, target visitor, and primary conversion action
- Required sections, copy, logo/images, and legal/contact information
- Preferred technology, delivery boundary, and launch destination

If an assumption materially changes the page, ask before implementation. Otherwise, label it in the History record.

## Page design checklist

- Put the main value proposition and CTA above the fold.
- Use proof only when it is supplied or verifiable.
- Keep navigation concise and every primary interaction keyboard accessible.
- Use Korean labels consistently; avoid untranslated framework jargon in visible UI.
- Test 375px, 768px, 1024px, and a desktop width.
- Respect reduced-motion preference and keep normal motion purposeful.
- Use text contrast of at least 4.5:1 for normal-size text.

## Verification record

Record the following in the project History file:

- Approved brief and assumptions
- Design-system choices and reasons
- Changed or created files
- Run/build command and outcome
- Browser/DOM checks, including CTA and navigation behavior
- Known limitations and the next improvement

## Experience entries

Add a dated entry only after a task is complete.

| Date | Context | Verified learning | Evidence | Reuse rule |
| --- | --- | --- | --- | --- |
| YYYY-MM-DD | Homepage type | Concrete decision that worked | Preview/test/file | When it applies |

Do not record credentials, private customer information, unverified marketing claims, or generic preferences without evidence.
